# Lucas Design Authority Protocol

Lucas é a autoridade final de produto, UX, design, escopo e prioridade.

Comentários `Lucas Review` no GitHub são instruções oficiais.

Comentários criados via Slack também são instruções oficiais.

Checks verdes não significam aprovação.

Build passando não significa aprovação visual.

Produto aprovado = Lucas aprovou explicitamente.

Se Lucas disser que o design está superficial, o PR não está aprovado.

Se Lucas pedir design novo, não entregar refresh de copy, borda, radius ou card.

Se Lucas pedir parar, parar.

Se Lucas pedir draft/open, não marcar ready.

Se Lucas pedir não fechar issue, não usar `Closes #`.

O agente deve ler os comentários mais recentes do PR antes de trabalhar.

Se existir `Status: não aprovado` ou `Status: bloqueado`, corrigir o PR atual antes de avançar.

## Qualidade Visual Esperada

O app deve parecer uma aplicação nova, moderna, clara, guiada e profissional.

Não aceitar:

- formulário gigante;
- tabela como experiência principal;
- card com texto longo;
- refresh superficial;
- copy bonita em tela antiga;
- mobile apenas empilhado;
- botão sem hierarquia;
- estados de pendência fracos;
- design interno com cara de ferramenta técnica bruta.

Aceitar:

- fluxos por etapa;
- decisão clara;
- CTA evidente;
- revisão humana;
- confiança e fonte visíveis;
- técnico sob demanda;
- mobile pensado para uso real;
- 3D como apoio visual útil;
- orçamento com status e fonte.
